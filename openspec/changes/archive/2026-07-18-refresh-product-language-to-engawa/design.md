## Context

現行UIでは「退避」が以下の異なる意味を兼ねている。

- 撮影後に item を作る行為: 「退避して、次を撮る」
- item 作成の完了: 「退避しました」
- 直後の完全撤回: 「退避を取り消す」
- 未決定のモノの説明: 「候補へ退避」

また、未決定の一覧は「候補」と呼ばれている。一方、`candidate` から `discarded` への変容を「手放す」、移り先を「捨離」と呼ぶ関係は世界観と機能が両立しているため維持する。

本changeはコピー層だけを刷新する。データモデルと画面フローは安定しているため、内部識別子を同時に変更しない。

## Goals / Non-Goals

**Goals:**

- 行為を「写しを収める」、未決定の居場所を「縁側」として役割ごとに呼び分け、既存の「手放す→捨離」は維持する。
- 初見でも「縁側」の意味を理解できる説明を、既存画面のコピー内で提供する。
- 開いた青円相と閉じた朱円相の意味を、新しい語彙と矛盾なく接続する。
- 実削除・実保存は詩的な言い換えで曖昧にせず、操作結果が分かる語を使う。
- 375x812 の既存レイアウトとアクセシビリティを保つ。

**Non-Goals:**

- `ItemStatus`, SQLite列、ルート、ファイル、コンポーネント、関数の内部名変更。
- 新しい初回フラグ、オンボーディング画面、説明モーダルの追加。
- 画面数・ナビゲーション・永続化・円相描画方式の変更。
- archive 済み仕様履歴の遡及修正。
- バックエンド・同期・クラウド保存の追加。

## Decisions

### 1. 行為・場所・変容を別の語で表す

語彙の文法を次のように固定する。

| 層 | canonical term | 意味 |
|---|---|---|
| コア原則 | `決めずに、写しを収める。` | 決断より先に写真と記録を確保する |
| ユーザー行為 | `写しを収める` | 写真を持つ item を作成する |
| 未決定の居場所 | `縁側` | 捨てる／残すをまだ決めないモノの一覧 |
| 手放す行為 | `手放す` | item を `discarded` へ移し、写真と記録を残す |
| 移り先・変容後 | `捨離` / `手放し済み` | `discarded` の一覧・状態 |

「写し」は物理的なモノそのものではなく写真と記録を指す。「縁側」はそのモノを表す item が身を置く比喩的な場所を指す。このため **「縁側に収める」へ一語化しない**。写しは「収める」、モノは縁側へ「置く／戻る」と書き分ける。

代替案:

- 「退避」を「縁側」へ機械的に置換する。→ 「縁側を取り消す」「縁側中」など不自然な文が生まれるため不採用。
- 「候補」だけを残す。→ 戻り導線と左タブで世界観が分断されるため、user-facing 表示は「縁側」に統一する。
- 内部 `candidate` も `engawa` へ変更する。→ UIコピー刷新に不要なDB移行と回帰リスクを生むため不採用。

### 2. UIコピーの対応表

実装時は文字列の一括置換ではなく、以下を基準に文脈ごとに更新する。

#### 縁側・空状態

| 場所 | copy |
|---|---|
| 左タブ | `縁側` |
| 空状態見出し | `捨てるか迷ったら、撮る。` |
| 空状態本文 | `捨てるでも、残すでもない。\nここは、モノの縁側です。` |
| 空一覧の短文 | `縁側にはまだ、モノがありません` |
| 絞り込みゼロ | `この条件のモノはありません` |

「縁側一覧」という複合語は建築物の一覧に読めるため使わず、導線は簡潔に「縁側へ」とする。

#### 撮影・写しを収める

| 場所 | copy |
|---|---|
| 主アクション | `写しを収めて、次を撮る` |
| 副アクション | `写しを収めて、書き留める` |
| 写真ゼロ | `まず、写真を一枚撮りましょう` |
| 処理中 | `収めています` |
| 失敗 | `写しを収められませんでした。` |
| 権限補足 | `ライブラリの写真から、写しを収めることもできます。` |
| 完了見出し | `写しを収めました` |
| 完了補足 | `このモノは、縁側へ` |
| 左一覧への導線 | `縁側へ戻る` |
| 直後の完全撤回 | `今の記録を取り消す` |

#### 手放す・捨離（既存語彙を維持）

| 場所 | copy |
|---|---|
| 詳細の主アクション | `手放す` |
| 確認見出し | `{name} を手放しますか？` |
| 確認本文 | 現行の手放し確認コピーを維持する |
| 確認の主アクション | `手放す（捨離リストへ）` |
| 確認の副操作 | `まだ、そばに置く` |
| 完了本文 | `{name} を手放しました。記録は「捨離」に残ります。` |
| 完了キャプション | `— 捨離リストへ移りました —` |
| 捨離への導線 | `捨離リストを見る` |
| 縁側への導線 | `縁側へ戻る` |
| 一覧見出し | `手放した記録` |
| 状態 | `手放し済み` |
| 復帰 | `もう一度、縁側へ` |
| 復帰注記 | `手放したことにしたけれど、まだ手元にある時に` |

完了演出の「円が、ひとつ結ばれました」と捨離詳細の「N日間そばに」は、円相とモノへの距離を表す既存の情緒コピーとして残す。

### 3. 「保存」「削除」は実際の操作に限って使う

新語彙は操作結果の明確さより優先しない。

- 端末フォトライブラリへ画像を保存する操作は `画像をライブラリに保存` のままとする。
- 写真ファイルを端末内のアプリ領域へ書き込めなかった技術エラーは「保存」を使用してよい。
- item と写真を完全に消す直後の撤回は `今の記録を取り消す` とし、捨離（記録を保持する変容）と区別する。
- 通常の item 作成を「保存」「削除」「退避」とは呼ばない。

### 4. 内部名と表示名を分離する

このchangeでは以下を維持する。

- `ItemStatus = 'candidate' | 'discarded'`
- `CandidateList`, `DiscardedList`, `EvacuationComplete`, `ReleaseConfirm`, `ReleaseComplete`
- `releasedAt` / `released_at`
- `releaseItem`, `restoreItem`, `releaseDraft`, `canRelease` など既存識別子

これらは今回の検索監査対象には含めるが、ユーザーに露出していないことを確認するためであり、rename対象ではない。内部用語の整理は、必要なら別changeに分離する。

### 5. 円相は「記録の完了」と「判断の未完」を同時に示す

- 青・開いた円相: 写しを収める行為は完了したが、捨てる／残すの判断は開いたまま。モノは縁側にいる。
- グレー円相: 手放しを確認している静かな時間。まだ状態は変わっていない。
- 朱・閉じた円相: 手放しが完了して捨離へ移り、円が結ばれた。

青円相にチェックを出すことは「決断完了」ではなく「写しを収めた完了」なので矛盾しない。

### 6. 説明は空状態と完了画面だけに置く

新しいオンボーディング状態は追加しない。空状態本文で「モノの縁側」を明示し、写しを収めた完了画面で「このモノは、縁側へ」と結果を示す。通常の縁側一覧には説明文を常設しない。既存データがあって空状態を見ない人も、完了画面とタブラベルから意味を学べる。

スクリーンリーダー向けには、左タブの accessibility label を `縁側、捨てるか残すかをまだ決めていないモノ` 相当の説明にする。視覚ラベルは短い `縁側` を保つ。

### 7. 検索監査は語彙ごとに扱いを分ける

実装時の主検索語:

- 置換候補: `退避`, `候補`
- 維持確認: `手放す`, `手放し`, `手放した`, `捨離`
- 文脈監査のみ: `保存`, `削除`, `変容`
- 非表示確認のみ: `Evacuation`, `Candidate`, `candidate`, `release`, `discarded`

対象は画面本文だけでなく、エラー、accessibility label、権限説明、テスト期待値、README、正本、active OpenSpec を含む。archive 済みchangeと `legacy/vue-pwa/` は履歴・参照実装として変更しない。

## Risks / Trade-offs

- **[「縁側」が初見で機能を示さない]** → 空状態の定義文、完了補足、説明的なaccessibility labelをセットで更新する。
- **[「写し」が複製一般に読める]** → 撮影直後の文脈とカメラアイコンを保ち、写真ゼロ時は平易な「まず、写真を一枚撮りましょう」を使う。
- **[詩的コピーで実削除が曖昧になる]** → 完全撤回だけは「今の記録を取り消す」とし、捨離との違いをspecで固定する。
- **[一括置換で不自然な文が生まれる]** → 対応表に基づいて文脈単位で変更し、旧語彙の残存確認後に人手で全件レビューする。
- **[375px幅でボタンが折り返す]** → 375x812で主・副ボタン、Dynamic Type、スクリーンリーダーを確認する。
- **[権限文言だけ旧語彙が残る]** → `app.json` を監査し、変更を反映したネイティブバイナリで確認する。
- **[正本と実装の語彙が再びずれる]** → `AGENTS.md`, design handoff, product spec, OpenSpec config, active specsを同じchangeで更新する。

## Migration Plan

1. 正本とactive specの語彙を更新し、コピー対応表を実装判断の基準にする。
2. 縁側／撮影／捨離の順にuser-facing文字列、エラー、accessibility labelを更新する。
3. `app.json` の権限説明を更新する。
4. 検索監査で旧語彙を分類し、許容箇所（内部識別子・archive・legacy・実保存）以外の残存を解消する。
5. typecheck/testと375x812 visual QAを行い、権限説明は新しいネイティブバイナリで確認する。

データ移行やDBロールバックは不要。コピー変更を戻す場合はUI文字列と正本のコミットを戻し、権限文言を戻したネイティブバイナリを再配布する。
