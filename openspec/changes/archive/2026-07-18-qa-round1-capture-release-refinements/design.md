## Context

初回実機QA（Expo dev build / iPhone）で、撮影〜手放しの主要フローに実機で触れて得たフィードバックをまとめて反映する。個々は小さいが、いくつかは既存 spec の思想（捨離詳細は読み取り専用）や依存関係（外部ライブラリ追加）に触れるため、実装前に方針を固める。

現状の関連実装:

- 撮影は `CaptureScreen` が担い、`useCaptureDraft` フックが撮影ドラフト（複数枚・撮り直し・破棄）を管理する。退避は `saveItem('capture'|'edit')` で確定し、写真0枚のときは内部で `takePhoto()` を呼ぶガードがある（データは壊れないが、退避ボタンが写真前から見えているのが混乱の元）。
- FAB は `BottomNav` から `navigation.getParent()?.navigate('Capture')` で撮影画面へ。撮影画面はカメラを自動起動せず、プレースホルダを表示する。
- 写真保存は `photoStorage` にあり、`captureAndStorePhoto`（カメラ）と `pickAndStorePhoto`（ライブラリ）が実装済み。後者はどの画面からも未接続。
- 手放し完了は `ReleaseCompleteScreen`。導線は「捨離リストを見る」のみ。退避完了 `EvacuationCompleteScreen` は「次を撮る」「候補一覧へ戻る」の両方を持つ。
- 捨離詳細 `DiscardedDetailScreen` は読み取り専用。記録編集 `ItemEditScreen` は候補向けで、日付はテキスト入力＋「今日」クイック入力。

## Goals / Non-Goals

**Goals:**

- 撮影フローを「撮ってから退避」の順序でUIに表現する（FAB即カメラ、退避アクションは写真1枚以上で提示）。
- 写真の追加ソースにライブラリ選択を加える（`pickAndStorePhoto` を接続）。
- 手放し完了から候補一覧へ戻れるようにする。
- 手放し済みの品名だけを編集可能にする。
- 記録編集の日付をカレンダーUIで選べるようにする。
- 迷い度アイコンの円相はみ出しを直す（視覚）。

**Non-Goals:**

- バックエンド・アカウント・クラウド同期の追加（ローカルファーストを守る）。
- 手放し済みの品名以外の項目編集。
- ギャラリー複数選択・ドラッグ並べ替え・撮り直しUIの刷新など体験拡張。

## Decisions

### 1. 撮影優先フロー（A/B）

- **FAB即カメラ**: 撮影画面 `Capture` に入ったら、`expo-camera` のアプリ内カメラを直接表示する。FAB→画面遷移→カメラ、で「余計なタップなし」を満たす。カメラ内にライブラリ導線を置き、Xで撮影をやめた場合は空のレビューへ留まらず候補一覧へ戻す。
  - 代替案: FAB から直接 `launchCameraAsync` を呼んでから画面へ。→ ナビゲーションと権限・キャンセル処理が分散するので不採用。撮影画面の責務に寄せる。
- **退避アクションのゲート**: `photos.length === 0` のとき「退避して〜」ボタン群を描画しない（＝ゲート）。写真が1枚以上になったら表示する。これで `saveItem` 内の0枚ガードは保険として残しつつ、UI上は撮影前に退避が押せない。
  - spec 上は「写真ゼロでは退避アクションが出ない」シナリオを capture に追加済み。

### 2. ライブラリ選択（D）

- 既存 `pickAndStorePhoto` を接続する。撮影画面の「もう1枚」と、詳細/編集の「写真を足す」に、カメラ/ライブラリの選択肢を出す。UIは軽量に：撮る導線の近くに「ライブラリから選ぶ」を並置（ActionSheet 相当は最小構成で、まずはボタン2択で可）。
- 保存・孤児ファイル削除の後処理は既存の追加パスと共通化する（DB追加失敗時に保存済みファイルを削除）。
- 撮影ドラフト（`useCaptureDraft`）にもライブラリ由来URIを流せるよう、写真取得を「カメラ or ライブラリ」で差し替え可能にする。ドラフトの破棄・撮り直しの後始末は既存のまま（取得元に依らずURIを一様に扱う）。

### 3. 日付ピッカー（E）

- `@react-native-community/datetimepicker` を採用。Expo SDK 57 managed 対応版を `npx expo install` で入れる（SDK互換の版が入る）。iOSはネイティブのカレンダー/ホイールUI。
  - 代替案: 自作カレンダー。→ 保守コスト高。標準ライブラリで足りる。
  - 代替案: `expo` 同梱の何か。→ 日付ピッカーの第一候補は community 版が定番。
- 値の真実は従来どおり `YYYY-MM-DD` 文字列。ピッカーの `Date` ⇄ `YYYY-MM-DD` 変換は純関数として `dateLabels`（既存）付近に置き、Vitest で検証する。「今日」クイック入力は残す。

### 4. 手放し完了の戻り導線（F）

- `ReleaseCompleteScreen` に「候補一覧へ戻る」を追加し、`navigation.replace('Tabs', { screen: 'CandidateList' })` へ。`EvacuationCompleteScreen` のボタン構成に合わせて視覚と文言を対称にする。副作用は持たない（spec通り）。

### 5. 手放し済みの品名編集（G）

- 思想整理: 手放し=変容で記録は残す。「呼び名の訂正」までは変容と矛盾しない、という立場を取る。よって**品名(name)だけ**を捨離詳細から編集可能にし、status・released_at・その他項目は不変とする。
- 実装: 既存 `updateItem` は name 更新を扱えるので流用する。捨離詳細に軽いインライン編集（タップで編集→保存）を足す。保存時に status を触らないことを保証する（`updateItem` は status を変更しない前提を明示・テスト）。
- spec: release「捨離の詳細を見る」を品名編集可へ改定し、「手放し済みの品名を編集する」を追加。item-record「記録を閲覧する」の読み取り専用を品名例外つきに改定。

### 6. 迷い度円相のはみ出し（C）

- `CaptureScreen` の `urgencyCircle` / `ensoRing` のサイズ関係を見直す。円相（EnsoMark 88 / EnsoImage 80）が枠を超えないよう、コンテナに `overflow: 'hidden'` は使わず（円相を切りたくない）、円相サイズと枠サイズ・余白を調整して収める。トークンは `tokens.ts` に寄せる方針を維持。純粋な視覚調整で spec 変更は無し（タスクのみ）。

### 7. カメラのズームと切替（H）

- `expo-camera` の `zoom` は 0〜1 の割合値で、上限まで使うと過剰な拡大になるため 0〜0.5 にクランプする（`CAMERA_ZOOM_MIN`/`CAMERA_ZOOM_MAX`、`src/lib/cameraZoom.ts`）。
- ピンチは `react-native-gesture-handler` で受け、`log2(scale)` ベースの感度係数で自然な効き方にする。不正なスケール（0以下・非有限）は開始時ズームへフォールバック。
- 1x/2x のプリセットボタンを置き、現在値との近接判定（±0.015）で選択状態を示す。前面/背面切替時はズームを 1x に戻す（前面カメラに 2x 相当が無いため）。
- 純ロジック（クランプ・ピンチ計算・プリセット判定）は Vitest で検証する。

### 8. 手放し済み写真のライブラリ保存（I）

- 手放し=変容の思想では記録はアプリ内に残るが、「手元を離れたモノの写真を端末にも残したい」は自然な要望のため、**手放し済み（discarded）に限って**端末ライブラリへの保存を許す。候補の写真は退避中の記録であり、外へ持ち出す導線は置かない。
- ネイティブは `expo-media-library`（権限リクエスト→ `Asset.create`）。権限拒否時は設定から許可を促す文言で失敗を知らせる。Web は `<a download>` によるブラウザダウンロードで代替。
- ファイル名は品名から生成し、不正文字はハイフンへ置換。品名が空になる場合は「不断捨離」をフォールバック名にする（`photoDownloadFileName`、Vitest 検証）。
- 導線は捨離詳細の写真ヒーロー右上の download と、全画面ビューアの3点メニュー（手放し済みのみ表示）。

### 9. ヘッダーの設定ボタン（J）

- 共通ヘッダー（`AppHeader`）右上の moon アイコンは機能を持たず紛らわしいため、設定ボタン（歯車）へ差し替える。設定項目が未定の間はタップで「設定項目は、これから追加予定です。」の準備中案内を出す（`onSettingsPress` で将来差し替え可能にしておく）。
- 設定画面そのものの設計は別 change とする（今回は入口の確保のみ）。

## Risks / Trade-offs

- **[FAB即カメラで権限拒否時にループ感]** → 権限拒否や撮影キャンセル時は自動再起動せず、プレースホルダに留めてユーザー操作を待つ。自動カメラは「空ドラフトでの初回マウント時のみ」に限定する。
- **[datetimepicker のSDK互換]** → `npx expo install` でSDK対応版を取得。managed prebuild 済みなので、ネイティブモジュール追加後は `expo prebuild`/再ビルドが要る点に注意（実機再インストール）。
- **[品名編集が「変容」の思想を薄める懸念]** → 編集対象を品名のみに厳格に限定し、status/日付/他項目は不変。UIでも「品名の訂正」に見えるよう最小の編集に留める。
- **[ライブラリ選択UIの追加でボタンが増える]** → まずは最小（カメラ/ライブラリの2択）で導入し、ActionSheet 化などは後日。

## Migration Plan

1. `npx expo install @react-native-community/datetimepicker` → `expo prebuild` → 実機再ビルド。
2. 純ロジック（退避ゲート条件、日付⇄文字列、写真ソース抽象）を helper 化し Vitest 追加。
3. 画面を順に更新（Capture → 完了画面 → 編集日付 → 写真追加ソース → 捨離品名編集）。
4. 各段階で実機QA。ロールバックはコミット単位で戻す（依存追加コミットは分離しておく）。

- D（ライブラリ選択）のUI形：まずボタン2択で入れ、ActionSheet/シート化は別changeにするか。
